## Migration architectural decisions (deferred to #2 / #3 brainstorming)

These are patterns that need decisions before the TS / Lit migrations land. Captured here so they're not re-discovered.

### Wire-methods-onto-svgCanvas pattern

Every `init()` function in `packages/svgcanvas/core/*.js` mutates the svgCanvas instance:

```js
let svgCanvas = null
export const init = (canvas) => {
  svgCanvas = canvas
  svgCanvas.clearSelection = clearSelectionMethod
  // ... 9 more method assignments
}
```

TS can't see these as members of `SvgCanvas`. Three resolutions to choose from at #2 kickoff:

- **(a)** `.d.ts` augmentation declaring all wired-on methods as `SvgCanvas` members
- **(b)** Refactor to actual instance methods on `SvgCanvas` class
- **(c)** Per-module `interface SvgCanvas { ... }` augmentation declarations

### jgraduate stack disposition (deferred to #3 brainstorming)

The 5 jgraduate files total ~4,737 lines. Three options:

1. **Lit-rewrite both jGraduate + jPicker.** Preserves gradient editor (irreplaceable for an SVG editor). Estimated 4,737 → ~1,500-2,500 lines.
2. **Replace with modern color-picker library** (e.g., vanilla-picker). Loses gradient editor — significant SVG-editor feature regression.
3. **Hybrid:** Lit-rewrite jGraduate (gradient editor), replace jPicker (solid-color) with a library. Mid-ground.

Recommendation lean: option 1 for fidelity, option 3 for scope reduction. **Strategic call at #3 kickoff.**

## The pathseg-drop transition plan

Per Area J corrected scope. The `PathDataListShim` in `path-method.js:45-200` already provides the bridge for `pathSegList` consumers — but the polyfill also provides `createSVGPathSeg*` constructor methods that the shim does NOT cover. 8 direct call sites in `path-actions.js` need refactoring.

### Steps

1. **Drop polyfill imports (7 sites):**
   - `packages/svgcanvas/svgcanvas.js:10` — `import 'pathseg'`
   - `tests/unit/path.test.js:2`
   - `tests/unit/path-actions.test.js:1`
   - `tests/unit/text-actions.test.js:1`
   - `tests/unit/draw.test.js:1`
   - `tests/unit/utilities-bbox.test.js:2`
   - `tests/unit/utilities-performance.test.js:2`

   - `package.json` `dependencies.pathseg`
   - `scripts/copy-static.mjs:22` — `['node_modules/pathseg/pathseg.js', 'tests/vendor/pathseg/pathseg.js']`

3. **Refactor 8 `createSVGPathSeg*` call sites in `path-actions.js` to use `setPathData` with plain `{type, values}` objects** (chosen over fallback pattern for cleaner long-term API uniformity):
   - Line 419: `createSVGPathSegLinetoAbs(absX, absY)`
   - Line 420: `createSVGPathSegCurvetoCubicAbs(...)`
   - Line 422: `createSVGPathSegClosePath()`
   - Line 474: `createSVGPathSegLinetoAbs(...)`
   - Line 475: `createSVGPathSegCurvetoCubicAbs(...)`

   - Line 1030: `createSVGPathSegClosePath()`
   - Line 1235: `createSVGPathSegLinetoAbs(lastM.x, lastM.y)` (inside `fixEnd` block adjacent to the `// Can this be done better?` comment at line 1237)
